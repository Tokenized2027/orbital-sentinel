Full Project Description

Sentinel by Orbital is an autonomous AI agent platform that monitors DeFi protocol health using Chainlink CRE workflows. It reads live Ethereum mainnet contract data, feeds it through AI risk analysis, and writes verifiable proof hashes on-chain. No human in the loop.

We built 8 production CRE workflows for stake.link, the largest Chainlink liquid staking protocol:

1. Treasury Risk: reads staking pool contracts (getTotalPrincipal, getMaxPoolSize, getRewardBuckets) via EVMClient, computes risk scores across 4 dimensions, calls Claude Haiku for structured assessment.

2. Governance Monitor: polls Snapshot GraphQL and Discourse forum via HTTPClient with consensusIdenticalAggregation, flags urgent votes with less than 24 hours remaining.

3. Price Feeds: reads Chainlink Data Feed contracts (LINK/USD, ETH/USD) via latestAnswer(), computes stLINK/LINK depeg in basis points.

4. Morpho Vault Health: reads Morpho Blue market utilization rates and ERC4626 vault TVL via EVMClient, flags high utilization (liquidity crunch risk).

5. Token Flows: tracks SDL and stLINK balances across 50+ classified addresses (NOPs, whales, DEX pools, vesting) via EVMClient, detects large movements indicating protocol stress.

6. CCIP Lane Health: monitors Chainlink CCIP lane availability by reading Router, OnRamp paused state, and LockReleaseTokenPool rate limiter buckets via EVMClient.

7. Curve Pool Monitor: reads the Curve LINK/stLINK StableSwap pool (balances, amplification factor, virtual price) via EVMClient, computes pool imbalance using LINK/USD Chainlink price feed.

8. LINK AI Arbitrage (LAA): monitors stLINK/LINK arbitrage opportunities via Curve pool. Reads pool balances, premium quotes at multiple swap sizes, Priority Pool queue status via EVMClient. Calls GPT for AI analysis of optimal swap timing. This workflow is live on the CRE mainnet DON, running 7x/day autonomously.

Every workflow run produces an immutable on-chain audit trail: a HealthRecorded event on SentinelRegistry (Sepolia), containing the keccak256 hash of workflow-specific metrics. Risk levels use a prefixed format (treasury:ok, feeds:warning, morpho:critical) so each proof is tagged with its source workflow.

A composite intelligence layer reads data from 5 workflows, combines it with the LAA arb signal, and feeds everything to AI as cross-workflow context. The composite assessment gets its own on-chain proof, creating verifiable records of ecosystem-aware AI reasoning.

On-chain proofs are written autonomously across all 8 workflows + composite, 7 cycles per day. The contract deduplicates identical hashes, so only changed assessments produce new records. The dashboard collects HealthRecorded events via viem getLogs, stores them in PostgreSQL, and surfaces them with per-workflow statistics and Sepolia Etherscan links.

How Is It Built?

All 8 workflows are built with @chainlink/cre-sdk (v1.0.9) using Runner, handler, CronCapability, EVMClient, HTTPClient, consensusIdenticalAggregation, getNetwork, and encodeCallMsg.

EVMClient.callContract() reads staking pools, reward vaults, Morpho Blue markets, Chainlink Data Feeds, CCIP infrastructure, Curve StableSwap pools, and ERC20 token balances directly from Ethereum mainnet contracts. Each workflow makes 10+ on-chain reads per run.

HTTPClient with consensusIdenticalAggregation handles off-chain data: AI analysis endpoints (Claude Haiku for treasury risk, GPT for arb analysis), Snapshot GraphQL for governance, and Discourse forum for proposal discussion. The consensus mechanism ensures oracle nodes agree on the response before accepting it.

SentinelRegistry.sol is a minimal Solidity contract (0.8.19) deployed on Sepolia for on-chain health proofs. Owner-gated writes, duplicate prevention via mapping, input validation. Audited with 32 tests and 80,000 fuzz iterations.

Proof hashes are computed in TypeScript using viem's keccak256 + encodeAbiParameters, identical encoding to Solidity's abi.encode(). Each workflow encodes domain-specific metrics into the hash.

The dashboard is Next.js 15 with Drizzle ORM reading from PostgreSQL. A collector reads HealthRecorded events from Sepolia via viem getLogs and surfaces them with CRE capability tags and Etherscan links.

The AI analysis server is a Flask endpoint (Python) running Claude Haiku for treasury assessment and GPT for arb and composite analysis, served through a stable Cloudflare tunnel with X-CRE-Secret auth header.

The LAA workflow is compiled to WASM via Javy, deployed to the CRE mainnet DON (Workflow Registry on Ethereum mainnet), and runs autonomously 7x/day.

What Challenges Did You Run Into?

Chaining EVM reads from mainnet contracts to an HTTP AI call to an EVM write on Sepolia within a single CRE workflow run. The data flow crosses chains and capabilities, and each step depends on the previous one's output.

Making AI analysis work with consensusIdenticalAggregation. AI responses are non-deterministic by nature, so we had to engineer prompts and response formats carefully to ensure oracle nodes reach consensus on the AI output. Temperature settings, structured JSON responses, and deterministic prompt templates were critical.

Managing wallet gas on Sepolia for continuous autonomous writes across 8 workflows and composite intelligence. We needed proper nonce management, gas estimation, and staleness dedup (the contract reverts on duplicate hashes via an on-chain mapping) to keep the system running without manual intervention.

Building the cross-workflow composite intelligence layer. Individual CRE workflows are isolated by design, so creating ecosystem-aware AI reasoning required a separate phase that reads all workflow snapshots and feeds combined context to the AI. The challenge was encoding multi-source metrics into a single verifiable on-chain proof.

Deploying to the CRE mainnet DON required compiling TypeScript workflows to WASM, setting up authenticated AI endpoints accessible from DON nodes (via Cloudflare tunnel), and managing CRE secrets for runtime configuration. The deploy pipeline (compile, upload, register on Workflow Registry) had to be debugged against the live DON environment.
